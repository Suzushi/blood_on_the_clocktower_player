## 1. 项目整体概述 (Project Overview)

**项目名称**: `blood_on_the_clocktower_K`  
**核心目标**: 构建一个《血染钟楼》（Blood on the Clocktower）桌面游戏的**在线玩家系统**。  
**应用形态**: Web 应用程序（Browser-based Tool）。  
**当前状态**: 可运行的 Web 原型，`暗流涌动` 已稳定可玩，`黯月初升` 为部分接入的扩展模块，`梦殒春宵` 仍处于数据先行、规则未闭合状态。

该系统旨在帮助说书人管理复杂的游戏状态，包括：

- 角色分配与设置。
- 夜间行动顺序引导。
- 信息生成（如占卜师、共情者等角色的信息反馈）。
- 白天的提名、投票与处决流程。
- 玩家状态追踪（存活/死亡、中毒、醉酒、保护等）。

## 3. 已完成内容 (Completed Features)

### 3.1 后端逻辑 (`main.py`, `game_data.py`)

- **游戏核心类 (`Game`)**: 已实现单局状态机，包括玩家管理、阶段流转（Setup -> Night -> Day）与主要胜负判定。
- **剧本数据**: 内置了三个官方剧本数据：
  - *Trouble Brewing* (暗流涌动)
  - *Bad Moon Rising* (黯月初升)
  - *Sects & Violets* (梦殒春宵)
- **角色分配算法**:
  - **随机分配**: 根据玩家人数自动计算板子（镇民/外来者/爪牙/恶魔数量），包含含特殊规则（如男爵、教父）的逻辑处理。
  - **手动分配**: 支持说书人完全自定义。
- **夜间结算引擎**:
  - **自动排序**: 根据 `night_order` 自动生成每晚的唤醒顺序。
  - **智能辅助**: 已为洗衣妇、图书管理员、调查员、厨师、共情者、占卜师、送葬者、钟表匠、侍女、女裁缝、筑梦师、神谕者、卖花女孩等角色接入信息生成或部分自动信息。
  - **状态处理**: 已接入击杀、保护、中毒、醉酒、复活、假死、替死等一批核心状态逻辑。
  - **已接入的 BMR 机制示例**: 驱魔人、恶魔代言人、赌徒、祖母、吟游诗人、主谋、僵怖、普卡、沙巴洛斯、珀、月之子、莽夫等。
- **白天流程**:
  - **提名与投票**: 已实现提名验证、投票统计、超时收票、最高票上板与房主强制结算。
  - **处决结算**: 已实现贞洁者、弄臣、恶魔代言人、和平主义者、主谋、圣徒、月之子等多条处决链路。

### 3.2 前端交互 (`player.html`, `player.js`, `style.css`)

- **统一玩家入口**: 通过 `/player` 完成建房、入座、查看身份、夜间行动、白天提名投票。
- **自动夜晚流程**: 由后端线程驱动顺序轮询，玩家按轮次提交自己的夜间选择。
- **消息系统**: 夜间结果、首夜恶魔/爪牙信息、守鸦人等结果通过玩家消息队列下发。
- **可视化状态**: 前端可展示存活、连接、投票、提名与公开日志等状态。

### 3.3 当前剧本支持结论

- **暗流涌动（TB）**: 当前主线剧本，已具备稳定可玩性。
- **黯月初升（BMR）**: 已有相当一部分角色接入主流程，但实现仍分散在 `main.py` 与 `player_api.py`，属于在建扩展模块。
- **梦殒春宵（SnV）**: 目前以角色数据和少量通用框架接入为主，核心规则未闭合，不应视为可正式开局的剧本。

## 4. 未完成内容与潜在缺憾 (Missing & Limitations)

1. **数据持久化 (Persistence)**:
    - 目前使用内存字典 `games = {}` 存储。
    - **风险**: 服务器重启或进程崩溃将导致**所有游戏进度丢失**。
    - *建议*: 引入 SQLite 或 JSON 文件存储。
2. **多端实时同步 (Real-time Sync)**:
    - 目前是传统的 HTTP 请求模式。虽然前端可能有轮询（未细看），但主要依赖操作触发更新。
    - **缺失**: 没有 WebSocket，状态一致性依赖轮询与请求返回。
3. **测试覆盖 (Testing)**:
    - 仓库中已经存在 `tests/`，但覆盖仍以回归补丁为主，尚未形成稳定的全链路测试矩阵。
4. **安全边界 (Auth / Trust Boundary)**:
    - 当前已采用“最小鉴权”模型，而不是账号系统。
    - `find_game` / `join_game` / `reconnect` 可匿名访问；玩家一旦入座，服务端签发 `reconnect_token`。
    - 所有玩家私有信息读取、提名、投票、夜间行动、消息读取、待处理行动等 `/api/player/*` 状态接口都必须携带该 token。
    - `game_id` 与 `player_id` 仅用于定位房间和座位，不再单独构成授权边界。
    - `owner_token` 继续承担房主管理权限，不扩展为账号体系。
    - 这套模型适合“点开即玩”的熟人局和轻部署场景；若要公网开放，仍应叠加 HTTPS 与更严格的入口保护。
5. **代码结构 (Maintainability)**:
    - `main.py` 仍承担大量规则、状态与结算逻辑，模块边界尚不清晰。
6. **剧本支持一致性**:
    - `game_data.py` 中已有大量 SnV 角色数据，但规则引擎并未同步闭合，容易形成“数据存在但不可正确游玩”的错觉。

## 5. 接下来的建议步骤 (Next Steps)

按照优先级推荐：

1. **接口收口 (Maintain Boundary)**:
    - 保持当前“入座即发 `reconnect_token`、玩家态接口强制校验”的边界。
    - 后续新增玩家能力优先落在 `/api/player/*`，不要再新增无鉴权的平行玩家入口。
2. **数据安全 (Data Safety)**:
    - 实现简单的文件持久化，每步操作后将 `Game` 对象序列化保存到磁盘，防止意外丢失。
3. **剧本矩阵治理 (Script Support Matrix)**:
    - 明确 TB / BMR / SnV 的“可玩 / 部分支持 / 数据保留”状态，并让前后端一致。
4. **代码重构 (Refactoring)**:
    - 持续将规则判定、信息生成、阶段推进从 `main.py` 拆出。
    - 将 `routes/game_routes.py` 中的兼容接口维持为薄转发层，不在兼容层堆叠新业务逻辑。
5. **功能增强 (Feature Enhancement)**:
    - **撤销功能 (Undo)**。
    - **自定义剧本**。
